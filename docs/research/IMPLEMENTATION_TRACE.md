# Implementation Trace: Research → Code

This file is the audit trail. For every claim the project makes about being "baked by research", there should be a specific research finding on one side and a specific piece of code on the other. The table below is that mapping. If a row is here, a skeptical reader can open the cited research file, open the cited code location, and decide for themselves whether the link is real.

No aggregate claims. No "informed by". Each row names one finding and one place the finding shows up in code or rules.

## What research we are NOT using yet

Honest gaps. These are real parts of the compendium that have no current reflection in the code or skill rules:

- **Category 03 (Persona & Character Design).** The project documents a voice-match procedure but does not ship persona vectors, a four-layer persona YAML, or any character-training-style versioned identity artifact. The `humanizer` skill is a subtractive layer, not a persona.
- **Category 10 (Style Transfer & Voice) beyond the manual voice-match checklist.** No authorship embedding, no TinyStyler/STAR-style small-expert model, no extract-then-apply pipeline. Voice match is a prompt-only procedure right now.
- **Category 19 (Agentic Autonomous Thinking) and Category 06 (Chain-of-Thought).** "Reason privately, humanize publicly" is recommended in the playbook but not wired in — the humanizer rewrites final output, it does not intervene on reasoning traces.
- **Category 20 (Memory & Personalization).** No style-memory block. Nothing persists across sessions. The Cat 20 "style memory as the sharpest unshipped primitive" finding is acknowledged in READMEs but not implemented.
- **Category 04 decoder-level levers (min-p, DRY, XTC, contrastive decoding).** The file rewriter targets text after generation. It does not touch the sampler stack, which Cat 04 names as the single largest humanization lever. That layer is out of scope for a skill that runs against existing model APIs.

## How to verify a claim in this file

For any row in the table:

1. Open the research file named in the "Source" column. Read the specific passage. The INDEX files at `docs/research/<NN>-<category>/INDEX.md` cross-cut five angle files each; the claim is usually stated in the INDEX summary and then elaborated in `A-academic.md` or `D-commercial.md`.
2. Open the code or skill file at the line range in "Where". Confirm the pattern, the rule, or the procedure is actually there. Line numbers may shift slightly as the files evolve — search for the named identifier (`SYCOPHANCY`, `_cap_em_dashes_per_paragraph`, etc.) if the line range is off.
3. Run the test that exercises it. `python3 -m pytest tests/humanizer-humanize/test_humanize.py -v` covers the regex rows (sycophancy, em-dash cap, stock vocab, hedging, transition tics, performative balance) and the preservation rows (code blocks, URLs, headings, YAML, blockquotes, tables, inline code). Rows that live only in the SKILL or README (warmth-reliability, anti-detector, detector false positives) have no unit test — verify them by reading the cited research and reading the cited section of `skills/humanizer/SKILL.md` or `README.md` side by side.

If a row doesn't hold up under all three steps, it's a bug in this file. Fix the row or delete it.
